ApiHug Copilot 基于 [Langchain4j](https://github.com/langchain4j/langchain4j/) 做模型接入，因此它的配置模型可以理解为三层：运行默认值、Vendor 档案和 Prompt 库。

`Toolbar > ApiHug > Settings > AI`

## 它是什么

设置页主要负责三件事：

- 定义聊天运行时默认参数
- 保存一个或多个 Vendor 配置
- 管理可复用的 Prompt 快捷指令

## 为什么重要

配置不清楚，通常会导致两类问题：

- 采样参数不稳定，回答质量起伏大
- 团队在云模型、私有代理和本地模型之间切换时上下文混乱

## 运行参数

`Toolbar > ApiHug > Settings > AI > Settings`

这些参数会作为正常对话的默认运行配置。

| 名称              | 作用说明                         |
|-------------------|----------------------------------|
| Chat Memory Size  | 最近消息保留窗口                 |
| Top-P             | 概率采样窗口                     |
| Max Output Tokens | 输出长度上限                     |
| Timeout(s)        | 请求超时时间                     |
| Max Attempts      | 重试次数                         |
| Temperature       | 回答随机性                       |
| Stream            | 是否流式输出                     |
| System Prompt     | 默认系统提示词                   |
| Show Detail       | 是否展示耗时和 token 细节        |

## Vendor 配置

`Toolbar > ApiHug > Settings > AI > Vendor`

Copilot 支持同一个供应商配置多个档案，这对区分开发、预发、生产或不同模型对比非常有用。

建议用直观的名字，比如 `OpenAI-dev`、`Azure-prod`、`Local-Ollama`。

常见供应商类型包括：

- OpenAI 风格 API
- Azure OpenAI
- Gemini
- Anthropic
- DeepSeek
- Qianfan
- Zhipu
- Ollama
- LM Studio
- Jan
- OpenRouter
- Tongyi 与 Hunyuan

### 档案公共字段

| 名称        | 说明                         |
|-------------|------------------------------|
| Name        | 在 UI 中显示的配置名         |
| Description | 可选说明，用来区分用途       |

### 常见连接字段

| 供应商类型    | 常见字段                      |
|---------------|-------------------------------|
| OpenAI 风格   | URL、API Key、可选模型名     |
| Azure AI      | endpoint、API Key、deployment |
| 本地运行时    | 本地 URL、可选模型名          |
| Qianfan       | API Key、Secret Key、Model    |
| Zhipu         | API Key、Model                |

只要某个服务提供兼容 OpenAI-style API 的接口，通常都可以通过自定义 Vendor 档案接入。

## Prompt 库

`Toolbar > ApiHug > Settings > AI > Prompt`

Prompt 快捷指令适合沉淀重复 ask，例如 `/explain`、`/review`、`/api`。

| 字段   | 说明                                   |
|--------|----------------------------------------|
| name   | slash 指令名，例如 `/explain`          |
| prompt | 这条快捷指令背后的完整提示词正文       |

Prompt 名称尽量保持简短、唯一、好识别。

### Prompt 编写建议

- 一条 prompt 只做一类清晰任务
- 需要项目约束时，直接把约束写进 prompt
- 对高频 ask，优先沉淀成 prompt 库，而不是每次手写长指令
- 团队共用的 prompt 最好可版本化管理
