ApiHug BMAD 把 BMAD 风格的 AI 交付方式落到 ApiHug 栈上，是需求、契约、实现与评审之间的工作流层。

## 它是什么

ApiHug BMAD 主要由三部分组成：

- 面向重复交付场景的工作流入口
- 理解 ApiHug 规则与边界的 agent
- 基于 REPL 的安装与更新方式

## 为什么重要

泛化 AI 工作流通常不知道 ApiHug 的契约在哪里、哪些层是生成的、评审时该看哪些规则。BMAD 的价值就是补上这层项目上下文。

## 如何安装

BMAD 依赖 [ApiHug REPL](/zhCN-docs/tool/apihug-repl)。

```shell
./apihug
apihug> install
```

常用安装方式：

```shell
install --directory ~/projects/my-api --ides cursor
install --quick-update true
install --compile-agents true
```

## 它如何融入整体流程

### 核心工作流

| Workflow 路径 | 主要用途 |
|---------------|----------|
| `/bmad:bmw:workflows:apihug-proto-design` | 生成或细化 protobuf 契约 |
| `/bmad:bmw:workflows:apihug-end-to-end` | 跑一条从需求到代码的完整链路 |
| `/bmad:bmw:workflows:apihug-project-review` | 按 ApiHug 规则审查项目 |
| `/bmad:bmw:workflows:apihug-app-implement` | 在契约已准备好的前提下实现应用逻辑 |

### 核心 Agent

`/bmad:bmw:agents:apihug` 是围绕 ApiHug 设计与评审场景的交互式 agent。

### 适用建议

- 重点是契约设计时，用 `apihug-proto-design`
- 需求横跨契约和代码时，用 `apihug-end-to-end`
- 需要规则驱动审计时，用 `apihug-project-review`
- 契约已经稳定、只剩实现时，用 `apihug-app-implement`
